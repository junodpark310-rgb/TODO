# Google Drive 수동 저장형 설계안

작성일: 2026-03-12

## 1. 결론

이 프로젝트에는 `자동 동기화`보다 `구글 로그인 + 필요할 때만 클라우드 저장/불러오기` 방식이 더 적합하다.

이유는 명확하다.

- 현재 앱은 이미 `localStorage` 기반으로 충분히 빠르고 안정적으로 동작한다.
- Google Drive를 항상 동기화 대상으로 쓰면 인증, 토큰 만료, 충돌 처리가 복잡해진다.
- 반대로 수동 저장 방식은 쓰기 횟수를 줄여 충돌 가능성을 낮춘다.

따라서 **로컬 저장은 그대로 유지하고, Google Drive는 선택적 백업 저장소로 추가**하는 것이 권장안이다.

## 2. 목표 재정의

이번 설계의 목표는 "실시간 멀티기기 동기화"가 아니다.

정확한 목표는 다음과 같다.

- 평소에는 각 기기에서 로컬로 빠르게 사용
- 사용자가 원할 때만 Google Drive에 JSON 저장
- 다른 기기에서는 사용자가 원할 때만 Google Drive에서 JSON 불러오기
- 저장 직전에 원격 변경 여부를 확인해 충돌을 줄이기

즉, 이 기능은 `클라우드 동기화`보다 **클라우드 체크포인트 백업**에 가깝다.

## 3. 권장 구조

### 기본 원칙

1. 앱의 평상시 저장소는 계속 `localStorage`
2. Google Drive는 선택 기능
3. Google Drive 저장은 자동이 아니라 사용자 액션으로만 실행
4. 클라우드 파일은 단일 JSON 파일 1개를 기본으로 운영

### 권장 저장소

1순위는 `appDataFolder`다.

- 앱 전용 숨김 저장소라서 구조가 단순하다
- 사용자가 파일을 실수로 옮기거나 수정할 가능성이 낮다
- "내 드라이브를 파일 저장소로 쓴다"는 요구는 충족하면서도 UX가 덜 복잡하다

2순위는 `drive.file` 기반의 보이는 JSON 파일이다.

- 사용자가 Drive에서 파일을 직접 보고 싶을 때만 선택
- 대신 파일 탐색, 폴더 관리, 파일 이동 같은 변수가 늘어난다

이번 설계안 기준 권장안은 **`appDataFolder` + 단일 JSON 파일**이다.

## 4. 사용자 흐름

### 첫 연결

1. 사용자가 `Google Drive 연결` 클릭
2. 구글 로그인 및 권한 승인
3. 앱이 기존 백업 파일을 찾음
4. 파일이 있으면 `불러오기` 여부를 묻고, 없으면 연결만 완료

### 클라우드 저장

1. 사용자가 로컬에서 계속 작업
2. 헤더의 `클라우드 저장` 버튼 클릭
3. 앱이 원격 파일 메타데이터를 먼저 확인
4. 충돌 없으면 JSON 업로드
5. 저장 완료 시 `마지막 클라우드 저장 시각` 갱신

### 클라우드 불러오기

1. 사용자가 `클라우드 불러오기` 클릭
2. 현재 로컬 데이터를 덮어쓸지 확인
3. 원격 JSON 다운로드
4. Zustand store에 반영
5. 로컬 저장소도 같이 갱신

## 5. 충돌 처리 전략

수동 저장 방식이면 충돌은 줄어든다.  
하지만 완전히 사라지지는 않는다.

예시:

- PC에서 수정 후 아직 클라우드 저장 안 함
- 모바일에서 예전 버전을 불러와 수정
- 두 기기 중 나중에 저장한 쪽이 클라우드 파일을 덮어쓸 수 있음

그래서 저장 직전에 아래 검사를 넣어야 한다.

### 저장 전 검사

- 마지막으로 불러왔던 원격 파일의 `modifiedTime`을 로컬에 기억
- 저장 버튼 클릭 시 현재 Drive의 `modifiedTime`을 다시 조회
- 값이 같으면 그대로 저장
- 값이 다르면 충돌 경고 표시

### 충돌 시 선택지

1. `클라우드 최신본 불러오기`
2. `내 로컬 내용으로 덮어쓰기`
3. `새 백업 파일로 저장`

초기 버전에서는 이 3개면 충분하다.

핵심은 **조용히 덮어쓰지 않는 것**이다.

## 6. 데이터 구조

현재 앱의 export 포맷을 그대로 재사용한다.

```ts
type CloudBackup = {
  version: '1'
  exportedAt: string
  tasks: Task[]
  timeboxes: Timebox[]
}
```

첫 버전부터 아래 필드는 추가하는 것이 좋다.

```ts
type CloudBackup = {
  version: '1'
  exportedAt: string
  deviceId: string
  tasks: Task[]
  timeboxes: Timebox[]
}
```

권장 이유:

- `exportedAt`: 어떤 쪽이 더 최신인지 판단
- `deviceId`: 나중에 충돌 메시지를 더 이해하기 쉽게 만들 수 있음

## 7. UI 설계

현재 헤더에 있는 `내보내기`, `가져오기` 옆에 아래 UI를 추가하면 된다.

- `Google Drive 연결`
- `클라우드 저장`
- `클라우드 불러오기`
- `마지막 저장 시각`
- `로컬 변경 있음` 표시

권장 UX 규칙:

- 로그인 안 된 상태에서는 `클라우드 저장`, `클라우드 불러오기` 비활성화
- 로컬 변경이 있으면 `로컬 변경 있음` 배지 표시
- 저장 성공 후 배지를 제거
- 불러오기 전에는 항상 덮어쓰기 확인 모달 표시

## 8. 상태 설계

새로 필요한 상태는 많지 않다.

```ts
type CloudState = {
  isConnected: boolean
  fileId: string | null
  lastSeenRemoteModifiedTime: string | null
  lastCloudSavedAt: string | null
  hasLocalChangesSinceCloudSave: boolean
}
```

역할:

- `fileId`: 같은 파일을 계속 업데이트하기 위해 필요
- `lastSeenRemoteModifiedTime`: 저장 직전 충돌 확인용
- `lastCloudSavedAt`: UI 표시용
- `hasLocalChangesSinceCloudSave`: 저장 필요 여부 표시용

## 9. 코드 설계

현재 코드 기준으로 아래 정도면 충분하다.

- `src/lib/cloudSerializer.ts`
  - store -> JSON
  - JSON -> store

- `src/lib/googleDriveAuth.ts`
  - 로그인
  - 액세스 토큰 요청

- `src/lib/googleDriveClient.ts`
  - 백업 파일 찾기
  - 파일 생성
  - 파일 다운로드
  - 파일 업데이트
  - 메타데이터 조회

- `src/stores/useCloudStore.ts`
  - 연결 상태
  - 마지막 저장 시각
  - 충돌 검사 관련 메타데이터

- `src/components/layout/CloudControls.tsx`
  - 연결/저장/불러오기 버튼 묶음

## 10. 동작 규칙

### 로컬 변경 감지

`tasks` 또는 `timeboxes`가 바뀌면 `hasLocalChangesSinceCloudSave = true`로 바꾼다.

이 값은 아래 상황에서 `false`로 바뀐다.

- 클라우드 저장 성공
- 클라우드 불러오기 완료

### 클라우드 저장 규칙

저장 버튼 클릭 시:

1. 로그인 상태 확인
2. 백업 파일 메타데이터 조회
3. `modifiedTime` 충돌 검사
4. 문제 없으면 JSON 업로드
5. 성공 시 `lastSeenRemoteModifiedTime` 갱신

### 클라우드 불러오기 규칙

불러오기 버튼 클릭 시:

1. 로그인 상태 확인
2. 원격 JSON 다운로드
3. 로컬 덮어쓰기 확인
4. store 반영
5. `lastSeenRemoteModifiedTime` 갱신

## 11. 왜 이 방식이 현재 앱에 맞는가

현재 앱은 문서 협업 도구가 아니다.  
실시간 동기화보다 다음 특성이 더 중요하다.

- 빠른 반응성
- 단순한 구조
- 적은 유지보수 비용
- 데이터 유실 방지

수동 저장형은 이 네 가지에 더 잘 맞는다.

특히 현재 코드에는 이미 JSON export/import가 있으므로, 클라우드 연동은 "새 저장소를 도입"하는 작업보다 **기존 백업 기능을 Google Drive에 연결하는 작업**에 가깝다.

## 12. 장점과 한계

### 장점

- 자동 동기화보다 충돌 가능성이 낮다
- 구현 범위가 작다
- 현재 `localStorage` 구조를 유지할 수 있다
- 오류가 생겨도 앱 기본 기능은 계속 동작한다
- 백엔드 없이도 구현 가능하다

### 한계

- 진짜 실시간 동기화는 아니다
- 여러 기기에서 동시에 수정하면 여전히 충돌 가능성이 있다
- 사용자가 저장 버튼을 눌러야 한다
- 불러오기 시 로컬 데이터 덮어쓰기 주의가 필요하다

## 13. 구현 순서

### 1단계

- export/import 로직을 공용 serializer로 분리
- Cloud 상태 store 추가
- 헤더에 클라우드 버튼 영역 추가

### 2단계

- Google 로그인 연결
- Drive 파일 검색/생성
- 수동 `클라우드 저장`

### 3단계

- 수동 `클라우드 불러오기`
- 저장 전 `modifiedTime` 충돌 검사
- 충돌 경고 모달 추가

### 4단계

- 새 백업 파일로 저장 옵션
- 마지막 저장 시각 UI 개선
- 오류 메시지 정리

## 14. 예상 일정

이 설계는 자동 동기화보다 범위가 작다.

- 기본 연결 및 저장: 1일
- 불러오기 및 충돌 검사: 0.5~1일
- UI 정리 및 테스트: 0.5~1일

총합:

- **빠르면 2일**
- **보통 2~3일**

## 15. 최종 권고안

이번 프로젝트에는 아래 조합이 가장 현실적이다.

1. 기본 저장소는 계속 `localStorage`
2. Google Drive는 선택 기능
3. 자동 동기화는 하지 않음
4. `Google Drive 연결 > 클라우드 저장 > 클라우드 불러오기`만 제공
5. 저장 직전 원격 `modifiedTime` 검사로 충돌 방지
6. 충돌 시 `불러오기 / 덮어쓰기 / 새 파일 저장` 중 선택

한 줄로 정리하면:

**이 앱에는 "실시간 동기화"보다 "수동 클라우드 백업"이 맞고, 그 구현은 충분히 현실적이다.**
